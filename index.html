Sell Blend tickets, natively inside Whish.

List Blend platform events, render tickets, waves, seat maps and questions, take the payment on Whish's own rail, and let Blend issue the ticket. One backend integration — Blend stays the source of truth for inventory, pricing and delivery.

The integration flow
01
Whish app → Blend

List events & the event page

GET /channel/events then /channel/events/:id — cards, tickets, waves, add-ons, custom questions, seating, all live from Blend.

02
Whish app → Blend

Show tickets & price

The event page returns live ticket prices (waves + fees already applied). Show them verbatim — pricing is finalized by Blend at checkout.

03
Whish app → Blend

Create the Whish payment

POST /channel/events/:id/checkout creates the order and opens a Whish Pay collect session, returning { collectUrl, externalId, orderId }.

04
Inside Whish

Buyer pays

The buyer settles the collect session natively with their Whish balance. No browser, no checkout page hosted by Blend.

05
Whish core → Blend

Blend confirms

Blend re-verifies the collect against Whish's status API, claims capacity, books held seats, and emails the ticket. Server-to-server — nothing for you to build.

06
Whish app → Blend

Deliver the ticket

Poll GET /channel/orders/status until paid: true, then show the pass and deep-link the QR at tickets.blendapp.ai/ticket/<id>.

01 — Core concepts

Blend owns the truth. Whish owns the payment.

Every read and write runs through the same engine that powers Blend's own checkout, so whatever an organizer changes on Blend appears in your API instantly. You never reimplement pricing, inventory, or fraud rules — you present them.

/ 01

Server-authoritative pricing

The amount you send must match Blend's computed total (±$0.01) or it's rejected as AMOUNT_MISMATCH. Always show the /quote numbers.

/ 02

One key, one organizer

A publishable key sees exactly one organizer's opted-in events. Platform-wide coverage = one key per organizer, merged in your backend.

/ 03

Whish is the rail

Create-payment opens a Whish collect session on Blend's account. The buyer settles it natively; Blend confirms only after Whish's status API says success.

/ 04

USD, end to end

The Whish rail is locked to USD across create, confirm, and refund. Amounts are major units (dollars), never cents.

02 — Quickstart

Your first paid ticket

Call the API from your backend. Every request carries your publishable key, an allowlisted Origin, and the real buyer IP so Blend's per-buyer rate limits key on the buyer, not your egress.

headers — every requestCopy
x-blend-key: <your Blend partner key>
x-blend-secret: <your Blend partner secret>
Content-Type: application/json
X-Forwarded-For: <the buyer's device IP>
the call sequence — paid GA ticketCopy
# 1 · list every platform event + open one
GET  /api/v1/channel/events
GET  /api/v1/channel/events/{id}

# 2 · quote — subtotal + Blend fee + total (show before pay)
POST /api/v1/channel/events/{id}/quote
     { "tickets":[{ "ticketId":"665f…", "quantity":2 }] }
     → { subtotal, fee, total, currency }

# 3 · create the order + open the Whish Pay collect
POST /api/v1/channel/events/{id}/checkout
     { "amount":49.5, "currency":"USD",
       "buyerName":"Rami K", "buyerEmail":"rami@example.com",
       "tickets":[{ "ticketId":"665f…", "quantity":2 }],
       "idempotencyKey":"whish-7f3a9c1b24d8" }
     → { orderId, externalId, collectUrl }

# 4 · buyer settles the collect in Whish, then poll
POST /api/v1/channel/orders/status
     { "externalId":1234567890123 }
     → { paid: true, ticket: { … } }
⚠︎ Idempotency keys must be globally unique

The 15-minute replay cache is keyed on the raw header value. Use a fresh random UUID per checkout attempt and reuse it only to retry the identical request — never derive it from cart contents or reuse it across buyers.

Free tickets skip the payment leg

A $0 order (free tier or a 100%-off code) goes straight to POST /channel/events/:id/checkout and confirms instantly with the QR email. Send the discount code in the body for the free path to apply.

03 — Always in sync

Change it on Blend, see it in Whish

There is no sync job on your side. Inventory is shared across every sales channel — a seat held in Whish is instantly unavailable on Blend's own site and at the box office.

05 — Reserved seating

Seat maps, held once and shared

When the event page returns seating.enabled: true, generate a stable X-Seat-Identity per buyer and send it on every seating call and on create-payment — the payment re-holds under that identity. Embed Blend's seat map as an iframe inside a host page, or render your own from the chart and drive the hold endpoints directly.

seated purchase — the identity threads throughCopy
GET  /channel/events/{id}/seats/config        # chart + prices
POST /channel/events/{id}/seats/hold          # X-Seat-Identity
POST /channel/events/{id}/checkout            # SAME X-Seat-Identity + holdToken
◆ One identity, everywhere

If the identity on the hold and the payment differ, the re-hold fails with 409 SEAT_UNAVAILABLE. Generate it once per purchase session and reuse it.

06 — Errors & limits

Fail predictably, retry safely

Errors are { success:false, code, message }. Match on code || error — a few endpoints carry the identifier in error. Rate limits are per-buyer on reads/writes; the burst, general and access-code limiters key on your egress IP, so agree an allowlist with Blend before a big on-sale.

CodeMeaningDo
AMOUNT_MISMATCHYour amount ≠ server totalRe-quote, re-present, retry with a new key
ORIGIN_NOT_ALLOWEDOrigin missing / not allowlistedSend an allowlisted Origin header
SOLD_OUTTier or event sold outRemove from sale, offer waitlist
waiting_room_requiredHigh-demand queue is activeRun the waiting-room flow first
SEAT_UNAVAILABLESeat lost or identity mismatchRefresh availability, re-select
EVENT_CANCELLEDEvent was cancelledBlock purchase, pull from catalog

Provision your key(s), agree the egress-IP allowlist, run the test matrix on staging, then ship. The full reference — every field, limit and error — is in the developer bundle.
